# Журнал работ на строительном объекте

Тестовое задание: внутренний инструмент для фиксации выполненных работ на строительном объекте. Прораб может вести ежедневный журнал, фильтровать записи по датам, добавлять, редактировать и удалять работы.

## Что реализовано

- Список записей журнала в табличном виде: дата, вид работ, объем с единицей измерения, исполнитель и комментарий.
- Фильтрация записей по диапазону дат.
- Добавление записи через форму с валидацией обязательных полей.
- Удаление записи.
- Редактирование существующей записи.
- Справочник видов работ и возможность добавить свой вид работ прямо из формы.
- Сводка по выбранному периоду: количество записей, общий объем, число исполнителей.
- Состояния загрузки, ошибок и пустого списка.
- Адаптивный интерфейс в стиле внутреннего B2B-сервиса: широкие окна, узкие окна, планшет и мобильный вид.


## Стек

- React 19
- TypeScript
- Vite
- Node.js
- Express
- SQLite через `better-sqlite3`
- Zod
- Vitest
- Supertest
- Playwright
- ESLint

##Почему 

Выбрал этот стек, потому что он хорошо закрывает требования тестового задания и при этом выглядит как нормальная база для внутреннего прод-инструмента.

React + TypeScript — потому что в задании это обязательное требование. TypeScript сразу дает типы для записей журнала, справочника видов работ, API-ответов и формы, поэтому меньше шансов сломать контракт между фронтом и бэком.

Vite — быстрый и простой сборщик для React-проекта. Для тестового задания он удобен: быстро запускается, без лишней конфигурации, хорошо работает с TypeScript и dev proxy на API.

Node.js + Express — простой и понятный backend для REST API. Здесь не нужен тяжелый фреймворк: CRUD, фильтрация по датам, валидация и работа с SQLite спокойно закрываются Express-ом.

SQLite + better-sqlite3 — потому что по заданию данные должны храниться в базе, не в памяти и не в файле. SQLite отлично подходит для тестового: база настоящая, хранится локально, не требует Docker/PostgreSQL, легко проверяется и переносится. better-sqlite3 выбрал за простую синхронную работу и надежность для небольшого внутреннего сервиса.

Zod — для серверной валидации входящих данных. Так API не принимает пустые даты, нулевой объем, пустого исполнителя или некорректный вид работ.

Vitest + Supertest — чтобы проверить бизнес-логику и API: создание, валидацию, фильтрацию, редактирование, удаление, справочник и пользовательские виды работ.

Playwright — для проверки реального пользовательского сценария через интерфейс: создать запись, отредактировать, удалить. Это ближе к тому, как приложение будет использоваться на практике.

ESLint — чтобы держать код аккуратным и ловить очевидные ошибки до сдачи.

```text
scripts/
  verify-browser.mjs  Проверка CRUD-сценария через браузер
```

## База данных

При запуске создается SQLite-база:

```text
data/work-journal.sqlite
```

Таблицы:

- `work_types` — справочник видов работ.
- `journal_entries` — записи журнала выполненных работ.

Папка `data/` добавлена в `.gitignore`, чтобы локальная база не попадала в репозиторий.

## API

```text
GET    /api/health
GET    /api/work-types
GET    /api/entries?from=YYYY-MM-DD&to=YYYY-MM-DD
POST   /api/entries
PUT    /api/entries/:id
DELETE /api/entries/:id
```
